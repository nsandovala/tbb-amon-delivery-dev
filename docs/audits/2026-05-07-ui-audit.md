```

POS dashboard → onSnapshot(tenants/tbb/orders, where createdAt >= startOfDay)
```

**Veredicto**: Los precios **se calculan en el backend** desde Firestore. No se trustan los precios del frontend. ✅

## Problema 1 — Carrito de tienda corta/oculta productos arriba

### Análisis del código

**Archivo**: `apps/web/src/components/cart/cart-drawer.tsx`

```
Drawer layout (mobile): max-md:h-[85svh] max-md:rounded-t-3xl
Drawer layout (desktop): md:right-0 md:top-0 md:bottom-0 md:w-[420px]

Body:
  div.flex.min-h-0.flex-1.flex-col         ← contenedor principal
    div.min-h-0.flex-1.overflow-y-auto     ← items (scroll)
    div.shrink-0.border-t                  ← summary (fijo abajo)
```

**Hallazgo**: La estructura flex es correcta (`min-h-0 + flex-1 + overflow-y-auto`). El contenedor de items debería poder scrollearse independientemente del summary. Sin embargo:

> **⚠️ PROBLEMA REAL**: El `CartSummary` component (341 líneas) contiene: nombre, teléfono, email, fulfillment toggle, dirección condicional, notas, método de pago (4 botones), resumen de totales y botón de confirmar. Todo eso va en la sección `shrink-0` (fija abajo). **En viewport móvil, el summary ocupa ~70% del drawer**, dejando un espacio mínimo para ver los items del carrito. Los productos se ven aplastados arriba, dando la percepción de que están "cortados".

### Severidad: 🔴 ALTA

### Evidencia visual

En el audit visual se confirmó que el cart drawer mobile tiene:
- Header (shrink-0): ~60px
- Items area (flex-1, overflow-y-auto): lo que sobra
- Summary (shrink-0): ~500px+ con formulario completo

**El formulario de checkout está DENTRO del summary del drawer**, no en un paso separado.

### Recomendación

- Separar el flujo de checkout en pasos: (1) revisar carrito → (2) datos del cliente → (3) confirmar.
- O hacer que el summary también sea scrolleable, convirtiendo el drawer completo en un scroll unificado.
- Alternativa mínima: hacer que el bloque del formulario colapse cuando hay items, mostrando solo el total y un botón "Ir a pagar" que expanda la sección.

## Problema 2 — Carrito/POS visualmente débil y se siente lento

### Análisis de performance

| Señal | Resultado |
|-------|-----------|
| `useTransition` / `startTransition` | ❌ No se usa en ningún lado |
| `Suspense` / lazy loading | ❌ No se usa |
| `useDeferredValue` | ❌ No se usa |
| Skeleton / loading states | ⚠️ Mínimos — solo "Cargando productos..." texto plano |
| `React.memo` en items pesados | ❌ No se usa en `CartItem`, `ProductCard` ni `OrderCard` |
| Image optimization | ⚠️ Web usa `next/image`; POS usa `<img>` nativo sin lazy |
| Cart persist deserialization | ⚠️ Zustand persist puede causar un flash en hydration |

**POS page**: 801 líneas en un solo archivo monolítico (`apps/admin/src/app/pos/page.tsx`). Todo el estado, handlers, rendering y side effects están en un solo componente. Cada keystroke en búsqueda causa re-render de toda la grilla de productos y del sidebar completo.

**Storefront cart**: El `useCartStore` se accede con selectores individuales, lo cual es bueno. Pero `cartItem` en `ProductCard` usa un selector que itera todo el array de items por cada card:

```ts
const cartItem = useCartStore((s) =>
  s.items.find((i) => i.product.id === product.id)
);
```

Con 9 productos y cada uno ejecutando `.find()`, hay 9 × N iteraciones por cada cambio de carrito.

### Severidad: 🟡 MEDIA

### Recomendación

- POS: Extraer en componentes separados (`PosProductGrid`, `PosCart`, `PosCustomerForm`, `PosDashboardKpis`).
- Storefront: Usar `React.memo` en `ProductCard` y `CartItem`. Considerar un Map en el store para lookup O(1) de items.
- Añadir skeletons para la carga inicial de productos.
- POS: Migrar `<img>` a `next/image` o añadir `loading="lazy"`.
- Implementar `useDeferredValue` para el campo de búsqueda del POS.

## Problema 5 — Totales del día, ventas y ticket promedio

### Análisis del código de KPIs (POS)

```typescript
// POS page — lines 254-266
const totalSalesToday = useMemo(() => {
  return todayOrders.reduce((acc, order) => {
    const value = order.totals?.total ?? order.total ?? 0;
    return acc + value;
  }, 0);
}, [todayOrders]);

const ordersTodayCount = todayOrders.length;

const averageTicketToday = useMemo(() => {
  if (ordersTodayCount === 0) return 0;
  return totalSalesToday / ordersTodayCount;
}, [totalSalesToday, ordersTodayCount]);
```

**Fuente de datos**: `todayOrders` viene de un `onSnapshot` con filtro `where("createdAt", ">=", startOfDay)`:

```typescript
const startOfDay = useMemo(() => {
  const now = new Date();
  now.setHours(0, 0, 0, 0);
  return Timestamp.fromDate(now);
}, []);
```

### Hallazgos

| KPI | Fuente | ¿Real o mock? | Notas |
|-----|--------|---------------|-------|
| Ventas hoy | `todayOrders.reduce(order.totals?.total)` | ✅ REAL | Lee de Firestore live |
| Pedidos hoy | `todayOrders.length` | ✅ REAL | Cuenta de documentos reales |
| Ticket promedio | `totalSalesToday / ordersTodayCount` | ✅ REAL | Cálculo correcto |
| `startOfDay` | `useMemo(() => ...)` | ⚠️ | Se fija al montar. Si el POS queda abierto >24h, el cálculo se congela |

**Veredicto**: Los KPIs son **datos reales de Firestore**, no mocks. ✅

#### Sin embargo:

1. **El `startOfDay` no se recalcula** — si el POS queda abierto después de medianoche, seguirá mostrando datos del día anterior como si fueran de hoy.
2. **No hay filtro de órdenes canceladas** — las órdenes canceladas suman al total del día, inflando ventas.
3. **Fallback `order.total`** — existe un fallback a `order.total` para backwards compat. Esto está bien pero implica que órdenes sin `totals.total` se leen por el campo legacy.

### En `/pedidos`

Los KPIs de `/pedidos` son diferentes: cuentan estados (en cola, preparando, listos, etc.) usando `orders.forEach()` agrupado. Son **datos reales live** sin mock. ✅

**Sin embargo**: No hay un KPI de ventas/revenue en `/pedidos`. Solo conteos de estado. El card "Moneda: CLP" es informativo estático.

### Severidad: 🟡 MEDIA

### Recomendación

- Recalcular `startOfDay` con un `useEffect` + `setInterval` cada 60 minutos.
- Excluir órdenes `cancelled` del cálculo de ventas y ticket promedio.
- Considerar agregar un KPI de revenue en `/pedidos`.

## Hallazgos adicionales (no reportados por usuario)

### H1 — Delivery fee inconsistencia frontend/backend

| Capa | Delivery fee |
|------|-------------|
| Storefront `CartSummary` | `fulfillmentType === "delivery" ? 1500 : 0` (hardcoded) |
| POS `page.tsx` | `fulfillmentType === "delivery" ? 1500 : 0` (hardcoded) |
| Backend `orders.service.ts` | `DEFAULT_DELIVERY_FEE = 1500` (server truth) |
| Backend `pos.service.ts` | `DEFAULT_DELIVERY_FEE = 0` (!!!) |

> **⚠️ BUG**: El POS frontend muestra $1.500 de delivery cuando se selecciona "Delivery", pero el backend POS (`pos.service.ts`) siempre calcula delivery como `$0`. El total que se muestra al usuario POS no coincidirá con el total real almacenado en Firestore.

**Severidad**: 🔴 ALTA — diferencia de monto visible al usuario vs almacenado.

### H2 — `createOrder` no envía `deliveryFee` desde el frontend

El API `createOrderApi()` en el web no envía `deliveryFee` en el payload:

```typescript
// web/src/lib/api/orders.ts
interface CreateOrderPayload {
  tenantId?: string;
  items: { ... }[];
  customer: { ... };
  fulfillmentType: "delivery" | "pickup";
  paymentMethod?: ...;
  // ← NO deliveryFee
}
```

Pero el backend acepta `deliveryFee` como optional y usa $1.500 por defecto. Esto funciona para "delivery" pero si fuera "pickup", el backend **igualmente cobra $1.500** porque no recibe el fulfillmentType para decidir. Veamos...

```typescript
// orders.service.ts line 86-94
export async function handleCreateOrder(
  tenantId: string,
  input: CreateOrderInput & { deliveryFee?: number }
): Promise<{ orderId: string }> {
  const { items, subtotal, delivery, total } = await calculateTotals(
    tenantId,
    input.items,
    input.deliveryFee ?? DEFAULT_DELIVERY_FEE  // ← siempre 1500
  );
```

> **⚠️ BUG**: Si un pedido web es "pickup", el frontend muestra $0 de delivery, pero el backend calcula $1.500. **El total almacenado en Firestore no coincide con lo que vio el cliente.**

**Severidad**: 🔴 ALTA

### H4 — Seed orders no usan la misma estructura que createOrder

El seed tiene:

```typescript
items: [{
  productId: "vader-burger",
  name: "Vader Burger",        // ← "name" en vez de "productName"
  qty: 1,
  unitPrice: 7990,
  subtotal: 7990,             // ← campo extra no usado

  subtotal: 9490,
  deliveryFee: 0,             // ← "deliveryFee" en vez de "delivery"
  discount: 0,                // ← campo extra

```

Pero `createOrder()` repo escribe:

```typescript
items: [{ productId, productName, qty, unitPrice, imageUrl, categoryId }]
totals: { subtotal, delivery, total }
```

Los nombres de campos difieren (`name` vs `productName`, `deliveryFee` vs `delivery`). Esto no causa errores porque el seed escribe directamente, pero la UI de pedidos usa `item.productId` + lookup para nombres, así que funciona. Sin embargo, el campo `totals.delivery` no existirá en órdenes seeded (solo `totals.deliveryFee`), y `pedidos/page.tsx` line 699 lee `selectedOrder.totals?.delivery`, que será `undefined` para órdenes del seed.

**Severidad**: 🟡 MEDIA

## Conclusión

El sistema está funcionalmente sólido post-E2E. Los flujos core (crear orden web, crear venta POS, cambiar estado, live stream) **funcionan y escriben datos reales en Firestore**. Los KPIs no son mocks — son cálculos live correctos.
